# Distilled: What Needs to Change for SFT-Ready Output

## The Core Problem

`trajectory.md` was designed for **human readability**, not for SFT training.
For SFT, we need the exact `(messages, tools, response)` triplet that the LLM API saw at each turn.

## Data Source

The agent debug JSON (`llm_debug/*_agent_response_*.json`) contains everything we need:
- `request.messages` -- full conversation history at final turn
- `request.tools` -- OpenAI-format tool definitions
- `response` -- the final assistant response

Since messages accumulate, we can slice the array to get per-turn training examples.

**Log mode caveat**: Default mode is `"latest"`, so only the FINAL agent call is saved. Earlier calls are overwritten. But the final call's `request.messages` contains all prior turns, so we can reconstruct everything.

## What to Build: SFT Data Extractor

Instead of converting to markdown, we need a script that:

### 1. Reads the debug JSON and extracts per-turn training examples

For each assistant message at index `i` in `request.messages`:
- **context** = `messages[0:i]` + `tools` + `tool_choice`
- **completion** = `messages[i]` (the assistant response)

### 2. Fixes the content array artifact

The `_format_messages_for_logging` function splits multi-line strings into arrays:
```python
# In the logged JSON, content may be an array:
"content": ["line1", "", "line2"]
# Must be reconstructed to the actual API value:
"content": "line1\n\nline2"
```

**Fix**: `"\n".join(content)` if content is a list, else use as-is.

### 3. Preserves ALL fields in the correct format

Each training example must include:

```json
{
  "messages": [
    {"role": "system", "content": "<string>"},
    {"role": "assistant", "content": "Hi! How can I help you today?", "tool_calls": null},
    {"role": "user", "content": "I'd like to exchange..."},
    {"role": "assistant", "content": null, "tool_calls": [
      {
        "id": "chatcmpl-tool-xxx",
        "name": "find_user_id_by_name_zip",
        "function": {"name": "find_user_id_by_name_zip", "arguments": "{...}"},
        "type": "function"
      }
    ]},
    {"role": "tool", "content": "yusuf_rossi_9620", "tool_call_id": "chatcmpl-tool-xxx"},
    ...
  ],
  "tools": [...],
  "completion": {
    "role": "assistant",
    "content": "...",
    "tool_calls": [...]
  },
  "reward": 1 or 0
}
```

### 4. Skips the hardcoded first message

The first assistant message `"Hi! How can I help you today?"` is injected by the Orchestrator (not generated by the LLM). Options:
- **Option A**: Skip it as a training target, but include it as context for turn 2+.
- **Option B**: Include it as a trivial training example (simple greeting).

Recommendation: **Option A** -- skip as target, keep as context.

### 5. Normalizes the response tool_calls format

The `response.tool_calls` in the debug JSON uses Tau2's internal format:
```json
{"id": "...", "name": "...", "arguments": {...}, "requestor": "assistant"}
```

This must be converted to litellm/OpenAI format:
```json
{"id": "...", "name": "...", "function": {"name": "...", "arguments": "<json-string>"}, "type": "function"}
```

## The 6 Critical Differences Between trajectory.md and SFT Needs

| # | What trajectory.md Does | What SFT Needs |
|---|------------------------|----------------|
| 1 | Drops `tool_calls[].id` | **Must preserve** -- correlates tool calls with results |
| 2 | Drops `tool_message.tool_call_id` | **Must preserve** -- API requires it |
| 3 | Omits `tools` array entirely | **Must include** -- defines what functions are available |
| 4 | Adds markdown formatting (headers, code fences, rules) | **Must output raw API format** -- no extra tokens |
| 5 | Pretty-prints tool args as `name({...})` | **Must keep as JSON string in `function.arguments`** |
| 6 | Renders as a single document | **Must produce per-turn (context, completion) pairs** |

## Recommended SFT Data Format

Output JSONL where each line is one training example:

```json
{
  "task_id": 0,
  "sim_id": "2c1ee543-...",
  "turn_idx": 3,
  "reward": 1,
  "messages": [...],
  "tools": [...],
  "completion": {...}
}
```

Where:
- `messages` = all messages up to (but not including) this assistant turn, with content arrays rejoined
- `tools` = the OpenAI-format tool definitions from `request.tools`
- `completion` = the assistant message at this turn (in litellm format)
- `reward` = 1 or 0 from evaluation results

For reward-prefix SFT, prepend the reward to the system prompt or use it as a special token.

## Effort Estimate

This is a focused script (~100-150 lines) that:
1. Reads the debug JSON
2. Fixes content arrays -> strings
3. Iterates messages to find assistant turns
4. Extracts (context, completion) pairs
5. Reads results.json for reward
6. Writes JSONL

No changes needed to tau2 itself. The debug JSON already has all the data; we just need to extract it correctly.

## Optional Enhancement: Change Log Mode to "all"

Currently `llm_log_mode` defaults to `"latest"`, keeping only the final agent call. If we change to `"all"`:

```python
set_llm_log_mode("all")
```

We'd get individual debug JSONs for each agent turn, each with its own `response` (including per-turn usage stats). But this is NOT required since the final JSON's messages array contains all turns.

The main benefit of "all" mode would be:
- Per-turn token usage stats (useful for analysis but not for SFT)
- Independent verification of reconstructed contexts
- The exact response object for each turn (vs reconstructing from messages)
